Recipe 04 — Shaders with shaderc

Author a shader once in bgfx's GLSL-like language; compile it with shaderc to Metal, SPIR-V and HLSL. The bgfx answer to the 3D track's SDL_shadercross story.

What you'll learn

  • bgfx's .sc shader dialect and how it differs from vanilla GLSL (mul, vec3_splat, SAMPLER2D, float-only uniforms).
  • varying.def.sc and the $input/$output contract between stages.
  • The shaderc command line — --type, --platform, -p profile, -i include path — and a build script that targets every backend.
  • The two ways to ship the result: load .bin files at runtime, or embed them with BGFX_EMBEDDED_SHADER.
  • bgfx's predefined uniforms (u_modelViewProj and friends) you get for free — the setup for recipe 06.

Prerequisites

The triangle from recipe 03, and a built shaderc binary (it falls out of the bgfx build under .build/<config>/bin/shadercRelease). You'll also need bgfx's shader include directory, bgfx/src, which holds bgfx_shader.sh.

Walkthrough

One language, every backend

You write shaders in a single GLSL-flavoured language. shaderc runs it through a C preprocessor and translates to whatever the target needs — MSL for Metal, SPIR-V for Vulkan, DXBC for Direct3D, GLSL/ESSL for OpenGL. The 3D track reached the same destination with SDL_shadercross; bgfx bakes the tool right into its ecosystem.

A few dialect rules bite newcomers — they exist because the source becomes HLSL under the hood:

  • Matrix math uses mul(m, v), never m * v.
  • Single-value vector constructors are vec3_splat(x); multi-value vec3(a,b,c) is fine.
  • Samplers use the SAMPLER2D(name, stage) macro (recipe 09).
  • Uniforms must be float-based — no bool/int uniforms.
  • Inputs/outputs are declared in varying.def.sc and pulled in with $input/$output.

The varying contract

varying.def.sc is a shared manifest: it names every vertex attribute and every inter-stage varying, with a semantic. Both shaders reference entries from it by name; the names in $input/ $output must match. 

# varying.def.sc
vec4 v_color0   : COLOR0;

vec3 a_position : POSITION;
vec4 a_color0   : COLOR0;

Compiling — the command

One shaderc invocation produces one .bin for one backend. The profile flag -p chooses the language; --platform sets preprocessor defines; -i points at bgfx's shader headers. 

# Metal (macOS)
shaderc -f vs_triangle.sc -o shaders/metal/vs_triangle.bin \
        --type vertex   --platform osx     -p metal \
        --varyingdef varying.def.sc -i $BGFX/src

# SPIR-V (Vulkan, Linux/Windows)
shaderc -f fs_triangle.sc -o shaders/spirv/fs_triangle.bin \
        --type fragment --platform linux   -p spirv \
        --varyingdef varying.def.sc -i $BGFX/src

# HLSL / Direct3D 11 (Windows ONLY — D3D shaders can't be cross-compiled off-Windows)
shaderc -f vs_triangle.sc -o shaders/dx11/vs_triangle.bin \
        --type vertex   --platform windows -p s_5_0 -O 3 \
        --varyingdef varying.def.sc -i %BGFX%/src

The output folder names (metal/, spirv/, dx11/, glsl/) match the shaderDir() switch in recipe 03, so the runtime loader picks the right binary for the active backend automatically.

Predefined uniforms come free. bgfx auto-fills a set of uniforms every frame: u_modelViewProj, u_model[], u_view, u_proj, u_viewProj and more. You set the matrices behind them with setTransform (model) and setViewTransform (view + projection) on the C++ side — no hand-rolled push constants like the 3D track. Recipe 06 uses u_modelViewProj for the first time.

Two ways to ship

Runtime load (recipe 03's loadProgram): ship the shaders/<backend>/*.bin tree next to your executable and read it with SDL_LoadFile. Simple, hot-swappable, but it's loose files on disk.

Embed: add --bin2c to each shaderc call to emit a C array instead of a binary, gather the per-backend arrays into one .bin.h, and let the BGFX_EMBEDDED_SHADER macro pick the right one at runtime. Now the shaders live inside the executable — a single self-contained binary, the bgfx equivalent of recipe 13's offline workflow in the 3D track.

Complete listing

The shaders (same files recipe 03 referenced)

# vs_triangle.sc
$input  a_position, a_color0
$output v_color0
#include "bgfx_shader.sh"
void main() {
    gl_Position = vec4(a_position, 1.0);
    v_color0    = a_color0;
}

# fs_triangle.sc
$input v_color0
#include "bgfx_shader.sh"
void main() {
    gl_FragColor = v_color0;
}

A build script for every backend

Drop this in examples/cpp-bgfx/shaders/. It compiles each .sc for every backend your platform can target. (On macOS you'll get Metal, SPIR-V and GLSL; DXBC needs Windows.) 

#!/usr/bin/env bash
# build_shaders.sh — compile every vs_*/fs_* for all local backends.
set -euo pipefail
SHADERC=${SHADERC:-shaderc}
BGFX_SRC=${BGFX_SRC:?set BGFX_SRC to your bgfx/src directory}

compile() {                     # $1=file  $2=type(vertex|fragment)
    local base="${1%.sc}"
    shaderc -f "$1" -o "metal/$base.bin"  --type "$2" --platform osx   -p metal  --varyingdef varying.def.sc -i "$BGFX_SRC"
    shaderc -f "$1" -o "spirv/$base.bin"  --type "$2" --platform linux -p spirv  --varyingdef varying.def.sc -i "$BGFX_SRC"
    shaderc -f "$1" -o "glsl/$base.bin"   --type "$2" --platform linux -p 440    --varyingdef varying.def.sc -i "$BGFX_SRC"
}

mkdir -p metal spirv glsl
for f in vs_*.sc; do compile "$f" vertex;   done
for f in fs_*.sc; do compile "$f" fragment; done
echo "shaders compiled."

Alternative: the embedded-shader loader

If you compiled with --bin2c, swap recipe 03's loadProgram for this — no files on disk, the right backend chosen at runtime: 

#include <bgfx/embedded_shader.h>
#include "vs_triangle.bin.h"   // generated: arrays vs_triangle_mtl, _spv, _dx11, _glsl ...
#include "fs_triangle.bin.h"

static const bgfx::EmbeddedShader kEmbedded[] = {
    BGFX_EMBEDDED_SHADER(vs_triangle),
    BGFX_EMBEDDED_SHADER(fs_triangle),
    BGFX_EMBEDDED_SHADER_END()
};

static bgfx::ProgramHandle loadProgramEmbedded(const char *vs, const char *fs) {
    bgfx::RendererType::Enum type = bgfx::getRendererType();
    return bgfx::createProgram(
        bgfx::createEmbeddedShader(kEmbedded, type, vs),
        bgfx::createEmbeddedShader(kEmbedded, type, fs),
        true /*destroy shaders with program*/);
}

Either loader returns a bgfx::ProgramHandle, so the rest of every recipe in this track is identical regardless of which you choose.

Try it yourself

  1. Run shaderc with no arguments to print the full flag list, then add --verbose to a compile and read the generated target source it prints.
  2. Introduce a deliberate error — write a_position * u_model instead of mul(...). shaderc fails with a clear message; that's the dialect enforcing HLSL-compatible math.
  3. Tint the fragment output: gl_FragColor = v_color0 * vec4_splat(0.5);. Recompile only the fragment shader and relaunch — the triangle dims, no C++ rebuild needed (runtime-load path).
  4. Add --bin2c to one compile and open the generated header. You'll see the per-backend byte arrays the BGFX_EMBEDDED_SHADER macro stitches together.
