bgfx Track

Twelve recipes that pair SDL3 with bgfx — a "bring your own engine", graphics-API-agnostic renderer that puts Metal, Vulkan, Direct3D and OpenGL behind one C++ API. SDL3 owns the window, events and timing; bgfx owns the pixels. You'll learn the one thing the 2D and 3D tracks never have to — how to pull a native window handle out of SDL3 and hand it to a third-party renderer — then build up to a lit, textured, instanced scene with shaders authored once and compiled to every backend.

How to run the code

Unlike the 2D and 3D tracks, bgfx is not part of brew install sdl3 — it's a separate library you build yourself. The usual route is to clone bgfx together with its siblings bx and bimg and build them with bgfx's GENie/make setup (make on macOS produces the static libs and the shaderc tool under .build/). A package manager works too — vcpkg install bgfx or the equivalent conan recipe.

Each recipe's Complete listing is a drop-in replacement for examples/cpp-bgfx/main.cpp in the parent template — replace that file's contents, then build with make run-bgfx from the repo root. That folder's Makefile fragment links libbgfx, libbx and libbimg alongside SDL3, the same way cpp-gpu/ links SDL3 on its own.

Shaders are compiled offline. bgfx does not accept shader source at runtime — you author .sc files and compile them with shaderc into per-backend binaries. Recipe 04 covers the whole workflow; recipes 03 and 05–12 assume the compiled shaders it produces live in examples/cpp-bgfx/shaders/.
